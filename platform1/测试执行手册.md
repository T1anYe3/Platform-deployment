## 1. 快速测试（5分钟）

部署完成后，先跑一遍快速验证：

```bash
cd platform1-docker

# Step 1: 确认所有容器 healthy
docker compose ps

# Step 2: 运行健康检查
docker compose run --rm health-check

# Step 3: 验证核心 API
curl -s http://localhost:9200 | head -20          # ES 应返回 JSON
curl -s http://localhost:5601/api/status | head -5  # Kibana 应返回状态
curl -sk https://localhost:8200/v1/sys/seal-status | head -10  # Vault 应返回密封状态

# Step 4: 手动触发一次数据采集
docker compose exec bridge python3 /scripts/bridge-minio.py
docker compose exec bridge python3 /scripts/bridge-nifi.py
docker compose exec bridge python3 /scripts/bridge-suricata.py
docker compose exec bridge python3 /scripts/bridge-vault.py

# Step 5: 检查 ES 索引
curl -s http://localhost:9200/_cat/indices?v | grep -v '^\.'

# 预期输出 4+ 行：suricata-alerts-*, vault-audit-*, minio-audit-*, nifi-logs-*
```

全部通过 → 继续全量测试。  
有失败 → 跳到 [常见问题](#5-常见问题)。

## 2. 全量测试（30分钟）

全量测试包含 4 个场景：安全传输 → 威胁检测 → 数据吞吐 → 审计完整性。

# 确认在 platform1-docker 目录

| platform1-minio | Up (healthy) |

### 2.3 场景 A：安全传输率测试

**目的**：验证平台各服务端口的 TLS 加密状态。

```bash
python scripts/security-benchmark.py --metric tls
```

**预期结果**：
- Vault (8200): TLS 已启用
- 安全传输率 >= 95%

**手动验证**：
```bash
# 逐个测试端口 TLS
openssl s_client -connect localhost:8200 -servername vault.sec.local </dev/null 2>/dev/null | grep "Verify return code"
# 预期：Verify return code: 0 (ok) 或 18 (self-signed)
```

### 2.4 场景 B：威胁检测测试

**目的**：发送攻击流量，验证 Suricata 检测 → ES 入库 → Kibana 可视化的完整闭环。

```bash
# Step 1: 确认 Suricata 在运行
docker compose ps suricata

# Step 2: 启动一个监听端口用于攻击目标（另开终端）
#        或者用 nifi/8443 作为 HTTP 目标
#        最简单的办法：用 Python 起一个临时 HTTP 服务
docker exec -d platform1-nifi python3 -m http.server 8080

# Step 3: 生成 5 万条攻击流量
#        让 Suricata 检测到这些流量
python scripts/generate-attack-traffic.py \
  --target 127.0.0.1 \
  --port 8080 \
  --num-attacks 50000 \
  --normal-ratio 0.3 \
  --delay 0.001

# Step 4: 等待 bridge-suricata 采集（60秒一轮）
echo "等待 Suricata bridge 采集..."
sleep 65

# Step 5: 验证 ES 中的告警
curl -s http://localhost:9200/suricata-alerts-*/_count
# 预期 count > 0

# Step 6: 按 SID 统计告警（验证 3 条规则都触发）
curl -s -X POST "http://localhost:9200/suricata-alerts-*/_search" \
  -H "Content-Type: application/json" \
  -d '{"size":0,"aggs":{"by_sid":{"terms":{"field":"alert.signature_id"}}}}' \
  | python -c "import sys,json; d=json.load(sys.stdin); [print(f'  SID {b[\"key\"]}: {b[\"doc_count\"]} 条') for b in d['aggregations']['by_sid']['buckets']]"

# Step 7: 运行检测率基准
python scripts/security-benchmark.py --metric threat
```

**预期结果**：
```
SID 9900101: ~6000 条 (sqlmap)
SID 9900102: ~6000 条 (Nikto)
SID 9900103: ~6000 条 (/etc/passwd)
威胁检测率 >= 90%
```

**浏览器验证**：
1. 打开 http://localhost:5601
2. Discover → Suricata IDS Alerts
3. 时间范围选 "Last 1 hour"
4. 应看到告警柱状图和详细记录列表

### 2.5 场景 C：数据吞吐量测试

**目的**：测量 NiFi→MinIO 管道在大数据量下的传输速度。

```bash
# Step 1: 确保 MinIO mc 已配置
docker compose run --rm minio-init

# Step 2: 将测试数据导入 MinIO bucket
# 方法1：直接通过 mc 上传（测试存储吞吐量）
time docker compose exec -T minio sh -c "
  mc alias set local http://localhost:9000 minioadmin ChangeThis-Local-123! >/dev/null 2>&1
  mc cp /test-data/bench-1g.bin local/raw-data/ 2>&1
"
# 记录耗时，计算吞吐量 = 1024MB / 耗时秒

# 方法2：如果 NiFi 流程模板已配置，通过 NiFi 管道测试
# （需要先在 NiFi UI 中创建 GetFile -> UpdateAttribute -> PutS3Object 流程）

# Step 3: 验证数据完整性
# 生成原始文件 hash
sha256sum test-data/bench-1g.bin > /tmp/orig.hash
# 从 MinIO 下载并对比
docker compose exec minio sh -c "
  mc alias set local http://localhost:9000 minioadmin ChangeThis-Local-123! >/dev/null 2>&1
  mc cp local/raw-data/bench-1g.bin /tmp/downloaded.bin 2>&1
  sha256sum /tmp/downloaded.bin
"
# 比对两个 hash 是否一致

# Step 4: 运行吞吐量基准测试
python scripts/security-benchmark.py --metric throughput
```

**预期结果**：
- 数据吞吐量 >= 10 MB/s
- 文件 hash 前后一致（无数据损坏）

### 2.6 场景 D：审计完整性测试

**目的**：验证 Vault 操作是否被完整记录到审计日志。

```bash
# Step 1: 获取 Vault token
VAULT_TOKEN=$(docker compose logs vault-init 2>/dev/null | grep "Root token:" | tail -1 | sed 's/.*Root token: //')
echo "Vault Token: $VAULT_TOKEN"

# Step 2: 执行一系列 Vault 操作
docker compose exec vault sh -c "
  export VAULT_ADDR='https://localhost:8200'
  export VAULT_TOKEN='$VAULT_TOKEN'
  export VAULT_CACERT='/vault/tls/root-ca.crt'

  # 写入 secret
  vault kv put -tls-skip-verify secret/test/benchmark value=test123

  # 读取 secret
  vault kv get -tls-skip-verify secret/test/benchmark

  # 加密
  vault write -tls-skip-verify transit/encrypt/project-data-key plaintext=$(echo -n 'benchmark-test' | base64)

  # 签发证书
  vault write -tls-skip-verify pki/issue/internal-service common_name=test.local ttl=1h
"

# Step 3: 等待 audit bridge 采集
sleep 65

# Step 4: 检查所有操作是否被审计
curl -s "http://localhost:9200/vault-audit-*/_count"
# 预期 count >= 4（对应上面 4 个操作）

# Step 5: 按操作类型统计
curl -s -X POST "http://localhost:9200/vault-audit-*/_search" \
  -H "Content-Type: application/json" \
  -d '{"size":0,"aggs":{"ops":{"terms":{"field":"request.operation","size":20}}}}' \
  | python -c "import sys,json; d=json.load(sys.stdin); [print(f'  {b[\"key\"]}: {b[\"doc_count\"]}') for b in d['aggregations']['ops']['buckets']]"

# Step 6: 运行审计基准
python scripts/security-benchmark.py --metric audit
```

**预期结果**：
- 审计覆盖率 = 100%（所有操作都被记录）
- 在 Kibana → Discover → Vault Audit Logs 中应能检索到上述操作

## 4. 测试指标解读

### 基准报告示例

```json
{
  "summary": {
    "total_benchmarks": 6,
    "passed": 6,
    "failed": 0,
    "pass_rate_pct": 100.0
  }
}
```

### 各项指标含义

| 指标 | good 值 | 说明 |
|------|---------|------|
| secure_transmission_rate | >= 95% | 平台服务端口使用 TLS 的比例 |
| threat_detection_rate | >= 90% | 3 条 Suricata 规则被触发的比例 |
| certificate_compliance_rate | 100% | TLS 服务使用平台 Root CA 签发证书的比例 |
| audit_coverage_rate | 100% | Vault 操作被审计日志记录的完整性 |
| event_ingestion_rate | >= 99% | ES 索引中有数据的索引占比 |
| data_throughput | >= 10 MB/s | NiFi→MinIO 管道的数据传输速度 |

### 评级标准

| 等级 | 通过数 | 说明 |
|------|--------|------|
| 优秀 | 6/6 | 平台全部安全能力达标 |
| 良好 | 5/6 | 基本达标，个别指标需优化 |
| 合格 | 4/6 | 核心能力达标，部分待改进 |
| 不合格 | < 4 | 需要排查部署问题 |

## 5. 常见问题

### Q: Suricata 没有告警数据

```bash
# 1. 确认 Suricata 在运行
docker compose ps suricata
docker compose logs --tail 20 suricata

# 2. 确认 eve.json 有内容
docker compose exec -T suricata sh -c "wc -l /var/log/suricata/eve.json 2>/dev/null || echo 'empty'"

# 3. 手动触发 bridge 采集
docker compose exec bridge python3 /scripts/bridge-suricata.py

# 4. 如果 Suricata 抓不到流量（WSL2/Docker Desktop 网络问题）
#    可以手动往 eve.json 注入测试告警来验证链路
docker compose exec -T suricata sh -c 'cat >> /var/log/suricata/eve.json << '"'"'EOF'"'"'
{"timestamp":"2026-05-11T12:00:00.000000+0800","event_type":"alert","src_ip":"192.168.1.100","dest_ip":"10.0.0.1","proto":"TCP","alert":{"signature":"Platform1 Suricata SQLMap scanner","category":"web-application-attack","severity":3,"signature_id":9900101,"rev":8}}
EOF'
```

### Q: Vault bridge 报 `audit.log not found`

```bash
# 确认 vault-data 卷挂载到了 bridge
docker compose exec bridge ls -la /vault/data/
# 应该能看到 audit.log
```

### Q: NiFi bridge 报 auth 错误

```bash
# 确认 NiFi 是 HTTPS 模式还是 HTTP 模式
curl -sk https://localhost:8443/nifi-api/access/config | head -5
# 如果是 HTTP 模式，检查 NIFI_URL 环境变量
docker compose exec bridge env | grep NIFI
```

### Q: ES 索引都是空的

```bash
# 手动触发所有 bridge
docker compose exec bridge sh -c "
  for b in suricata vault minio nifi; do
    echo \"=== bridge-\$b ===\"
    python3 /scripts/bridge-\$b.py
  done
"

# 再检查索引
curl -s http://localhost:9200/_cat/indices?v
```

### Q: 端口被占用

```bash
# 查看哪些端口被占用（Windows）
netstat -ano | findstr "8200 9200 5601 9000 9001 8443"

# 修改 docker-compose.yml 端口映射，例如：
#   "8200:8200" 改为 "18200:8200"
```
